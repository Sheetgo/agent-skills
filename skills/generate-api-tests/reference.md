# Generate API Tests — Reference

Heavy reference for templates, formats, and expression syntax. Read this file when generating tests or running init.

## Expression Syntax (go-runner / expr-lang)

| Pattern | Syntax | Example |
|---------|--------|---------|
| Environment var | `${{ env.VAR }}$` | `${{ env.MAIN_FILE }}$` |
| Simple env substitution | `${VAR}` | `${MAIN_FILE}` |
| Variable from previous test | `${{ vars.id }}$` | `${{ vars.connection_id }}$` |
| Response status | `${{ response.Status }}$` | Capitalized (Go struct field) |
| JSON response field | `${{ json.field }}$` | `${{ json.values.id }}$` |
| Fixtures data | `${{ fixtures.key }}$` | `${{ fixtures.users.admin.email }}$` |
| Loop data | `${{ loopData }}$` | Current iteration value |
| String concatenation | `+` operator | `${{ vars.prefix + "-suffix" }}$` |
| Array access | `[index]` | `${{ json.items[0].name }}$` |
| Nil check | `!= nil` | `${{ json.data != nil }}$` |
| Empty string check | `!= ""` | `${{ json.id != "" }}$` |
| Array length | `len()` | `${{ len(json.items) > 0 }}$` |
| Type conversion | `string()`, `int()` | `${{ string(loopData) }}$` |

**Key differences from JavaScript:** `response.Status` (not `response.status()`), `env.VAR` (not `process.env.VAR`), `!= nil` (not `!== undefined`), `len(arr)` (not `arr.length`).

**Fixtures:** Loaded from `resorce.json` in the spec file's directory (note: filename preserves a legacy typo). Returns a JSON map accessible via `${{ fixtures.path.to.value }}$`. See CLAUDE.md Fixtures System section for details.

## Framework Detection Signals

| Framework | Detection Signals |
|-----------|-------------------|
| Flask | `from flask import`, `@app.route`, `Blueprint`, `Flask(__name__)` |
| FastAPI | `from fastapi import`, `@app.get`, `@router.post`, `APIRouter` |
| Django | `urls.py` with `path()`, `views.py`, `from django.http` |
| Express | `express()`, `app.get(`, `app.post(`, `router.get(` |
| NestJS | `@Controller`, `@Get()`, `@Post()`, `@Injectable()` |
| Go (stdlib) | `http.HandleFunc`, `http.ListenAndServe` |
| Go (Gin) | `gin.Default()`, `gin.New()`, `r.GET(`, `r.POST(` |
| Go (Echo) | `echo.New()`, `e.GET(`, `e.POST(` |

**Detection priority:** Check `package.json` → `go.mod` → `requirements.txt`/`pyproject.toml` → scan source imports → ask developer if ambiguous.

## Init: Entry Points to Scan

- Python: `app.py`, `main.py`, `__init__.py`, `wsgi.py`
- Node: `index.js`, `app.js`, `server.js`, `main.ts`
- Go: `main.go`, `cmd/*/main.go`

## Init: Parameter Extraction

- Path params: `/users/{id}`, `/users/:id`
- Query params: `request.args.get()`, `req.query`
- Body params: `request.get_json()`, `req.body`
- Types: from type hints, decorators, or validation

## Config File Format

**Always use folder includes** (`!include module/`), not individual file includes.

**Empty (init only):**
```yaml
# Generated by /generate-api-tests init
# Project: {{project-name}}
# Run tests: ./go-runner run --config spec-files/{{project-name}}-config.yaml

endpoints:
  # Tests will be added with /generate-api-tests create
  # - name: MODULE_NAME
  #   mode: default
  #   requests: !include module_name/
```

**With tests:**
```yaml
endpoints:
  - name: AUTH
    mode: default
    requests: !include auth/

  - name: USER
    mode: default
    requests: !include user/
```

**After generating tests, update `.api-spec.md`:**
1. Tested column: `No` → `Yes (YYYY-MM-DD)`
2. Coverage table: recalculate module and total counts
3. Coverage bar: update percentage and fill bar proportionally

**Coverage bar rendering:** Use `█` for filled and `░` for empty, 20 chars total. Formula: filled = round(percentage / 5). Examples:
- 0%: `░░░░░░░░░░░░░░░░░░░░` **0%**
- 25%: `█████░░░░░░░░░░░░░░░` **25%**
- 50%: `██████████░░░░░░░░░░` **50%**
- 75%: `███████████████░░░░░` **75%**
- 100%: `████████████████████` **100%**


## Output Structure

```
spec-files/
  {{module}}/{{METHOD}}_{{RESOURCE}}.yaml
  {{project-name}}-config.yaml
```

POST /connections → `spec-files/connection/CREATE_CONNECTION.yaml`

## Naming Conventions

- **File:** `METHOD_RESOURCE.yaml` (uppercase, underscores). GET=GET, POST=CREATE, PUT=UPDATE, PATCH=PATCH, DELETE=DELETE.
- **Test name:** `ENDPOINT_NAME - SCENARIO`
- **Scenarios:** SETUP, HAPPY PATH, MISSING X, EMPTY X, INVALID TYPE X, RESOURCE NOT FOUND, QUOTA EXCEEDED, UNAUTHORIZED, FORBIDDEN WRONG OWNER, CONFLICTING PARAMS, TEARDOWN

## Deep Endpoint Analysis Checklist

Trace the full code path before writing any test YAML.

### Step 1: Read View/Route Handler
Identify: all params (path, query, body, headers), required vs optional, early returns, controller calls, data transformations.

### Step 2: Read Controller/Service Layer
Follow every function call. For each: trace `if/else` branches (each = test case), `try/except` blocks (each exception = test case), validation logic, authorization checks, business rules (quotas, limits, flags), database lookups that can fail.

### Step 3: Map All Exceptions
Read imports, find every exception class. For each: what triggers it, what response it produces. Create at least one test per exception.

### Step 4: Map Response Variations
Document every distinct response shape: different success responses, different error codes, edge cases (empty lists, null fields).

## YAML Test Template

Each test file MUST start with a header documenting analyzed code paths:

```yaml
# Generated by /generate-api-tests create
# Endpoint: METHOD /path
# Source: module/file.py:line
#
# Code paths analyzed:
#   1. Happy path - primary flow (line XX)
#   2. Happy path - alternative flow (line YY)
#   3. ExceptionClass (controller.py:ZZ)
#   4. AuthorizationCheck (authorization.py:AA)

# === SETUP ===
- name: ENDPOINT_NAME - SETUP
  method: post
  path: /setup-endpoint
  body:
    name: "Test Fixture"
  vars:
    fixture_id: ${{ json.values.id }}$
  tests:
    - name: Setup successful
      assert: ${{ response.Status == 200 }}$

# === HAPPY PATH: Primary flow ===
- name: ENDPOINT_NAME - HAPPY PATH PRIMARY
  method: METHOD
  path: /endpoint-path
  body:
    required_param: ${{ env.REQUIRED_PARAM }}$
    resource_id: ${{ env.EXISTING_RESOURCE_ID }}$
  vars:
    result_id: ${{ json.values.id }}$
  tests:
    - name: Response status
      assert: ${{ response.Status == 200 }}$
    - name: Success status
      assert: ${{ json.status == 1 }}$
    - name: Has expected field
      assert: ${{ json.values.id != "" }}$

# === HAPPY PATH: Alternative flow ===
- name: ENDPOINT_NAME - HAPPY PATH WITH NEW SUB_RESOURCE
  method: METHOD
  path: /endpoint-path
  body:
    required_param: ${{ env.REQUIRED_PARAM }}$
    new_sub_resource:
      name: "Created via test"
  vars:
    result_id: ${{ json.values.id }}$
    sub_resource_id: ${{ json.values.sub_resource.id }}$
  tests:
    - name: Success status
      assert: ${{ json.status == 1 }}$
    - name: Sub-resource created
      assert: ${{ json.values.sub_resource != nil }}$

# === PARAM VALIDATION: Missing required_param ===
- name: ENDPOINT_NAME - MISSING REQUIRED_PARAM
  method: METHOD
  path: /endpoint-path
  body: {}
  tests:
    - name: Error status
      assert: ${{ json.status == 0 }}$

# === PARAM VALIDATION: Empty required_param ===
- name: ENDPOINT_NAME - EMPTY REQUIRED_PARAM
  method: METHOD
  path: /endpoint-path
  body:
    required_param: ""
  tests:
    - name: Error status
      assert: ${{ json.status == 0 }}$

# === PARAM VALIDATION: Invalid type ===
- name: ENDPOINT_NAME - INVALID TYPE NUMERIC_PARAM
  method: METHOD
  path: /endpoint-path
  body:
    required_param: ${{ env.REQUIRED_PARAM }}$
    numeric_param: "not_a_number"
  tests:
    - name: Error status
      assert: ${{ json.status == 0 }}$

# === BUSINESS RULE: Resource not found ===
- name: ENDPOINT_NAME - RESOURCE NOT FOUND
  method: METHOD
  path: /endpoint-path
  body:
    resource_id: "non_existent_id_000"
  tests:
    - name: Error status
      assert: ${{ json.status == 0 }}$

# === BUSINESS RULE: Quota exceeded ===
# NOTE: Needs specific setup to trigger quota limits
# - name: ENDPOINT_NAME - QUOTA EXCEEDED
#   ...

# === AUTH: No token ===
- name: ENDPOINT_NAME - UNAUTHORIZED
  method: METHOD
  path: /endpoint-path
  auth: none
  body:
    required_param: "value"
  tests:
    - name: Returns unauthorized
      assert: ${{ response.Status == 401 }}$

# === AUTH: Wrong owner ===
# NOTE: Requires second test user token in env (OTHER_USER_TOKEN)
# - name: ENDPOINT_NAME - FORBIDDEN WRONG OWNER
#   ...

# === CROSS-PARAM: Conflicting params ===
- name: ENDPOINT_NAME - CONFLICTING PARAMS
  method: METHOD
  path: /endpoint-path
  body:
    required_param: ${{ env.REQUIRED_PARAM }}$
    resource_id: ${{ env.EXISTING_RESOURCE_ID }}$
    new_sub_resource:
      name: "Should this be ignored?"
  tests:
    - name: Verify behavior
      assert: ${{ json.status == 1 || json.status == 0 }}$

# === TEARDOWN ===
- name: ENDPOINT_NAME - TEARDOWN
  method: delete
  path: /cleanup-path/${{ vars.result_id }}$
  tests:
    - name: Cleanup successful
      assert: ${{ response.Status == 200 }}$
```

## CI Pipeline Generation Details

### Step 1: Detect Cloud Provider

Scan project root for deployment files:

| File Pattern | Cloud |
|-------------|-------|
| `cloudbuild*.yaml` | GCP Cloud Build |
| `buildspec*.yml` | AWS CodeBuild (future) |
| `azure-pipelines*.yml` | Azure DevOps (future) |
| `.github/workflows/*.yml` | GitHub Actions (future) |

If multiple found or none found, ask user via AskUserQuestion.
Currently supported: **GCP only**. For other clouds, inform user and stop.

### Step 2: Analyze Existing Deploy File

Read the deploy YAML and extract:
- **Build steps**: Docker image name, build args, registry URL
- **Deploy config**: service name, region, port, memory, CPU, max instances, concurrency, timeout
- **Networking**: VPC connector, egress settings
- **Security**: service account, CMEK key, auth settings
- **Environment**: env vars, env files
- **Substitution variables**: all `${_VAR}` patterns (these become the pipeline's substitutions)

### Step 3: Ask Gap-Filling Questions

Ask via AskUserQuestion (one at a time):
1. Which deploy step/service to use as base? (if multiple services in deploy file)
2. go-runner Docker image URL? (e.g., `gcr.io/project/go-runner-image`)
3. Worker pool name? (or "none" for default)
4. Pipeline timeout? (default: 3600s)

### Step 4: Generate Pipeline File

Output: `cloudbuild-{project-name}-api-testing.yaml` in project root.

Use the GCP Cloud Build Testing Pipeline Template below as the base. Fill in all `{{placeholders}}` from the analyzed deploy file and user answers.

Pipeline structure (GCP):
1. **Build image** — same Docker build but with temp name `tmp-{service}-$BUILD_ID`
2. **Push image** — push to same registry
3. **Deploy temp instance** — `gcloud run deploy tmp-{service}-${_DEPLOYMENT_ENV}` with same config, captures URL to `/workspace/service_url.txt`
4. **Warmup** — health check loop (curl every 5s, timeout 5min, accepts 200/401/403/404)
5. **Run tests** — `BASE_URL=$$SERVICE_URL go-runner run --config ./spec-files/{project}-config.yaml --verbose`
6. **Cleanup** — `gcloud run services delete tmp-{service}-${_DEPLOYMENT_ENV} --quiet` with `waitFor` referencing test step

Include `timeout` and `options.pool` if worker pool specified.

**Important:** On GCP Cloud Build, if a step fails, subsequent steps don't run by default. The cleanup step uses `waitFor` to reference the test step ID, but cleanup only runs if the test step completes (pass or fail exit). For guaranteed cleanup, note in a YAML comment that users may want a separate cleanup trigger or use `--allow-failure` patterns.

## GCP Cloud Build Testing Pipeline Template

Parameterized template for `cloudbuild-{project-name}-api-testing.yaml`. Fill `{{placeholders}}` from the analyzed deploy file and user answers. Remove blocks marked `{{OPTIONAL: ...}}` if not applicable.

```yaml
# Generated by /generate-api-tests ci
# Project: {{project-name}}
# Based on: {{source-deploy-file}}
# Run: gcloud builds submit --config cloudbuild-{{project-name}}-api-testing.yaml

steps:
  # === BUILD ===
  - name: 'gcr.io/cloud-builders/docker'
    id: 'Build Image'
    args: ['build', '-t', 'gcr.io/$PROJECT_ID/tmp-{{service-name}}-$BUILD_ID'{{build-args}}]
    # {{build-args}} expands to: , '--build-arg', 'KEY=VALUE' per arg from deploy file

  # === PUSH ===
  - name: 'gcr.io/cloud-builders/docker'
    id: 'Push Image'
    args: ['push', 'gcr.io/$PROJECT_ID/tmp-{{service-name}}-$BUILD_ID']

  # === DEPLOY TEMP INSTANCE ===
  - name: 'gcr.io/google.com/cloudsdktool/cloud-sdk'
    id: 'Deploy Temp Instance'
    entrypoint: 'bash'
    args:
      - '-c'
      - |
        gcloud run deploy tmp-{{service-name}}-${_DEPLOYMENT_ENV} \
          --image gcr.io/$PROJECT_ID/tmp-{{service-name}}-$BUILD_ID \
          --region ${_REGION} \
          --allow-unauthenticated \
          --port {{port}} \
          --service-account ${_SERVICE_ACCOUNT_EMAIL} \
          {{OPTIONAL: --vpc-connector ${_VPC_CONNECTOR_NAME} \}}
          {{OPTIONAL: --vpc-egress {{egress-setting}} \}}
          --timeout {{timeout}} \
          --cpu {{cpu}} \
          --memory ${_MEMORY_SIZE} \
          --max-instances {{max-instances}} \
          --concurrency {{concurrency}} \
          {{OPTIONAL: --set-env-vars "^;^{{env-vars-string}}" \}}
          {{OPTIONAL: --key projects/$PROJECT_ID/locations/${_REGION}/keyRings/${_CRYPTO_KEY_RING}/cryptoKeys/${_CRYPTO_KEY_NAME} \}}
          --project $PROJECT_ID \
          --labels cost-center=${_DEPLOYMENT_SCOPE},environment=${_DEPLOYMENT_SCOPE},service=lbl-testing-{{service-name}}-${_DEPLOYMENT_SCOPE} \
          --format 'value(status.url)' > /workspace/service_url.txt

  # === WARMUP ===
  - name: 'gcr.io/google.com/cloudsdktool/cloud-sdk'
    id: 'Warmup Instance'
    entrypoint: 'bash'
    args:
      - '-c'
      - |
        export SERVICE_URL=$(cat /workspace/service_url.txt)
        echo "Warming up $$SERVICE_URL (Timing out in 5 minutes)..."

        SUCCESS=false
        for i in {1..60}; do
          STATUS=$(curl -s -o /dev/null -w "%{http_code}" "$$SERVICE_URL")

          # Accept 200 (OK), 401/403 (Auth required), 404 (Not found)
          # All prove the server is UP and responding
          if [[ "$$STATUS" =~ ^(200|401|403|404)$ ]]; then
            echo "Instance is READY (Status: $$STATUS)"
            SUCCESS=true
            break
          fi

          echo "Attempt $i/60: Not ready (Status: $$STATUS). Retrying in 5s..."
          sleep 5
        done

        if [ "$$SUCCESS" = false ]; then
          echo "ERROR: Instance failed to respond within 5 minutes."
          gcloud run services describe tmp-{{service-name}}-${_DEPLOYMENT_ENV} --region ${_REGION}
          exit 1
        fi

  # === RUN TESTS ===
  - name: '{{go-runner-image}}'
    id: 'Run API Tests'
    {{OPTIONAL: env:}}
    {{OPTIONAL:   - 'ENV_VAR=${_SUBSTITUTION}'}}
    entrypoint: 'sh'
    args:
      - '-c'
      - |
        export SERVICE_URL=$(cat /workspace/service_url.txt)
        echo "Starting tests against: $$SERVICE_URL"

        BASE_URL=$$SERVICE_URL go-runner run --config ./spec-files/{{project-name}}-config.yaml --verbose

  # === CLEANUP ===
  # NOTE: On GCP Cloud Build, if 'Run API Tests' fails, this step will NOT run
  # by default. For guaranteed cleanup, consider a separate scheduled cleanup job
  # or a Cloud Build failure notification trigger.
  - name: 'gcr.io/google.com/cloudsdktool/cloud-sdk'
    id: 'Cleanup Temp Instance'
    entrypoint: gcloud
    args: ['run', 'services', 'delete', 'tmp-{{service-name}}-${_DEPLOYMENT_ENV}', '--region', '${_REGION}', '--quiet']
    waitFor: ['Run API Tests']

timeout: {{pipeline-timeout}}s
{{OPTIONAL: options:}}
{{OPTIONAL:   pool:}}
{{OPTIONAL:     name: {{worker-pool-name}}}}
```

### Template Placeholders

| Placeholder | Source | Example |
|------------|--------|---------|
| `{{project-name}}` | From `.api-spec.md` or project dir name | `node-api` |
| `{{service-name}}` | From deploy file service name | `node-api` |
| `{{source-deploy-file}}` | Filename of analyzed deploy file | `cloudbuild.yaml` |
| `{{build-args}}` | `--build-arg` flags from deploy build step | `, '--build-arg', 'TAG_NAME=$TAG_NAME'` |
| `{{port}}` | `--port` from deploy file | `8000` |
| `{{timeout}}` | `--timeout` from deploy file | `1800` |
| `{{cpu}}` | `--cpu` from deploy file | `1` |
| `{{max-instances}}` | `--max-instances` from deploy file | `2` |
| `{{concurrency}}` | `--concurrency` from deploy file | `20` |
| `{{egress-setting}}` | `--vpc-egress` from deploy file | `all-traffic` |
| `{{env-vars-string}}` | `--set-env-vars` value from deploy file | `ENV_FILES=${_ENV_FILES};ENV_ROOT=${_ENV_ROOT}` |
| `{{go-runner-image}}` | User answer (Step 3 Q2) | `gcr.io/project/go-runner-image` |
| `{{pipeline-timeout}}` | User answer (Step 3 Q4, default 3600) | `3600` |
| `{{worker-pool-name}}` | User answer (Step 3 Q3, omit block if "none") | `projects/proj/locations/region/workerPools/pool` |

### Substitution Variables

Copy all `${_VAR}` patterns from the source deploy file. Common ones:

| Variable | Purpose |
|----------|---------|
| `${_DEPLOYMENT_ENV}` | Environment suffix (e.g., `staging`, `prod`) |
| `${_REGION}` | GCP region (e.g., `us-central1`) |
| `${_SERVICE_ACCOUNT_EMAIL}` | Cloud Run service account |
| `${_VPC_CONNECTOR_NAME}` | VPC connector for private networking |
| `${_MEMORY_SIZE}` | Memory allocation (e.g., `512Mi`) |
| `${_DEPLOYMENT_SCOPE}` | Cost center / environment label |
| `${_CRYPTO_KEY_RING}`, `${_CRYPTO_KEY_NAME}` | CMEK encryption |
| `${_ENV_FILES}`, `${_ENV_ROOT}` | Environment config paths |
